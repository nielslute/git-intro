Oplossing: Lokaal versiebeheer

Je kunt je een oplossing voorstellen waarbij op jouw computer verschillende versies van bestanden in een systeem opgeslagen worden, dit heet lokaal versiebeheer. Bijvoorbeeld dagelijks automatische backups in timepage maar ook de undo via cmd+z.

Als je per ongeluk een fout maakt kan je altijd een stapje terug naar een eerdere versie. Maar hoe werk je samen?

Oplossing: Gecentraliseerd versiebeheer

Je kunt je ook een oplossing voorstellen waarbij op één server verschillende versies van bestanden opgeslagen worden, dit heet gecentraliseerd versiebeheer. Iedereen verstuurd zijn bestanden naar die ene repository en daar blijven alle wijzigingen bewaard. Implementaties van dit principe zijn Subversion en CVS maar ook Git kan op deze manier gebruikt worden.

Als iemand per ongeluk een fout maakt kan iedereen altijd een stapje terug naar een eerdere versie. Maar wat als de server omvalt?

Oplossing: Gedistribueerd versiebeheer

Je kunt je ook een oplossing voorstellen waarbij op elke computer alle verschillende versies van bestanden opgeslagen worden, dit heet gedistribueerd versiebeheer. Elke computer kloont de volledige historie van alle bestanden en mappen in een repository. Iedereen werkt in de lokale kloon en deelt alle wijzigingen met iedereen. Het systeem zorgt voor samenvoegen, de juiste versies etcetera. Implementaties van dit principe zijn Mercurial en natuurlijk Git.

Git is een gedistribueerd versiebeheersysteem

Een van de tofste dingen van Git is dat het gedistribueerd is. Zodra je samenwerkt zijn er automagisch backups van alle bestanden!

Gecentraliseerd werken in Git

Zoals net is aangegeven kan je gecentraliseerd werken in Git. Voor kleine projecten is dit prima, het verschil met SVN is dat elke developer in Git een volledige kopie (git clone) van de hele repository heeft. Na het maken van aanpassingen wordt de aanpassing gereed gemaakt voor verzenden (git commit) en naar de server verstuurd (git push).

Integrating Manager, werken in Git

Een andere manier van werken die veel wordt toegepast in open source ontwikkeltrajecten is de Integrating Manager benadering. Één eindverantwoordelijke verstuurd bestanden (git push) naar een zgn. blessed repository.

Ontwikkelaars klonen die repository (git clone), maken lokaal aanpassingen, zetten die klaar (git commit), versturen die naar hun eigen publieke repository (git push) en vragen vervolgens de eindverantwoordelijke de aanpassingen binnen te trekken (git request-pull), samen te voegen (git merge), klaar te zetten (git commit) en te versturen (git push) naar de blessed repository.

Dit zie je veel bij open-source op GitHub.

Dictator en luitenanten, werken in Git

Voor enorme projecten is een ontwikkeltraject als bij de Linux kernel vaak het meest effectief. Binnen dit model zijn luitenanten de baas over specifieke subsystemen van een project.

Luitenanten zorgen voor het binnen trekken (git pull) en samenvoegen (git merge) van alle aangeboden (git request-pull) aanpassingen van ontwikkelaars binnen die specifieke subsystemen.

De dictator kijkt énkel naar door luitenanten aangeboden veranderingen (git request-pull) en trekt die vervolgens uit de repositories van luitenanten binnen (git pull) en stuurt ze naar de blessed repository (git push ).

Git commando's op een rijtje...

git clone: maken van een exacte kopie, een kloon, van een remote repository (kort: repo) naar een lokale repo.

git commit: klaarzetten van een aanpassing.

git push: versturen van aanpassingen van een lokale repo naar een remote repo.

git pull: binnentrekken van aangepaste bestanden uit een remote repo, soms als reactie op een pull-request.

git request-pull: iemand anders vragen door jou gemaakte veranderingen op te nemen in haar repo.

git merge: samenvoegen van verschillende versies van bestanden bij conflicten, bijvoorbeeld als twee ontwikkelaars tegelijk hetzelfde bestand hebben aangepast.
